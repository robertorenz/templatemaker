myPdfSign

A self-contained signed-PDF reader for Clarion — open a digitally-signed PDF and report who signed it straight out of the embedded PKCS#7 / CMS signature, in pure Clarion. Signer certificate, issuing CA, signing time, and a tamper hint. No DLL, no external library, no network.

PKCS#7 / CMS · DER ASN.1 reader · signer identity + integrity hint · one global object · pure Clarion

Overview #

myPdfSign adds one global object to your application — a PdfSignClass instance you can reach from any procedure or embed — and through it open a signed PDF and read the authoritative signer identity that lives inside the document's PKCS#7 signature.

It is a sibling of myCompress, myQRDraw and myBarcodeGen: a self-contained ANSI Clarion class that does real work with no outside dependency. Given a signed PDF it finds the signature dictionary, hex-decodes the embedded PKCS#7 / CMS SignedData blob, and walks the DER with a tiny ASN.1 reader to surface the signer certificate's Subject, the issuing CA, and the signing time — plus the human-readable intent fields the signer typed and a whole-file integrity check.

Install & register #

  1. Copy templates/myPdfSign/PdfSignClass.inc and PdfSignClass.clw to a folder on the Clarion redirection path (your app folder, or \clarion12\libsrc\win). They must be stored in ANSI, not UTF-8.
  2. Copy templates/myPdfSign/myPdfSign.tpl into your template folder.
  3. In the IDE: Setup ▸ Template Registry ▸ Register, pick myPdfSign.tpl.
  4. On the application's global Extensions, add myPdfSign - Global signed-PDF reader (once). Set the object name (default PdfSig).
That's the whole setup The class's LINK attribute pulls PdfSignClass.clw into the build automatically; the ONCE include guard means it is safe even if another template also pulls it in. The object is multi-DLL aware out of the box.

Multi-DLL handling

The reader instance is global data, so — exactly like myCompress — the template makes it multi-DLL aware through ABC's %DefaultExternal / %DefaultExport symbols:

  • Defined in the single-EXE build or the root DLL (PdfSig  PdfSignClass).
  • EXTERNAL (imported) in every other DLL/EXE that uses it (…,EXTERNAL,DLL(dll_mode)).
  • Exported from the root DLL so those imports resolve.

The class methods are non-VIRTUAL and the CLASS carries an unconditional LINK, so each target links its own copy of PdfSignClass.clw — only the shared instance needs exporting.

Quick start #

From any embed, call the global object (named in the prompts; PdfSig below). This is the example shipped in the template's instructions:

  IF PdfSig.ReadFile('contract.pdf')
    IF PdfSig.Signed
      Message('Signed by ' & CLIP(PdfSig.SubjectCN) &|
              '|e-mail : '   & CLIP(PdfSig.SubjectEmail) &|
              '|issued by '  & CLIP(PdfSig.IssuerCN) &|
              '|signed at '  & CLIP(PdfSig.SignTime) &|
              '|reason : '   & CLIP(PdfSig.Reason) &|
              '|intact : '   & CHOOSE(PdfSig.CoversWholeFile=1,'yes','NO'))
    ELSE
      Message('This PDF is not signed.')
    END
  END

ReadFile returns 1 when it parsed the file (look at Signed to learn whether a signature was actually present, and at ErrText if anything went wrong). After a successful read every result property is filled in.

Report() — the canonical identity block PdfSig.Report() returns the same key=value block the .NET oracle publishes as ground truth (see The .NET oracle). It is handy for logging, diffing, or showing the full extracted identity at a glance.

Reading from memory #

If the PDF is already in a memory buffer — pulled from a BLOB, a download, or another in-memory source — parse it directly with Read instead of ReadFile:

buf    &STRING                                  ! the PDF bytes already in memory
bufLen LONG                                     ! the REAL byte count
  CODE
  ! …load buf / bufLen from a BLOB, queue, or download…
  IF PdfSig.Read(buf, bufLen) AND PdfSig.Signed
    Message('Signed by ' & CLIP(PdfSig.SubjectCN) &|
            '|organization : ' & CLIP(PdfSig.SubjectO) &|
            '|unit : '         & CLIP(PdfSig.SubjectOU))
  END
Pass the real length — never rely on LEN() A Clarion STRING is fixed-length and space-padded; it does not remember how many bytes are "real", and a PDF is binary. Always pass the exact byte count you loaded as pLen — if trailing spaces creep in, the /ByteRange whole-file check (and the DER walk near the end of the buffer) will be wrong.

Read parses the buffer in place and does not take ownership of it. ReadFile, by contrast, allocates and owns its own copy of the bytes (freed by the destructor or on the next read).

Startup self-test #

Run a one-line smoke test once when the program starts. SelfTest() parses a tiny built-in signed PDF and returns 1 when the parser produced the expected identity:

  IF ~PdfSig.SelfTest()
    Message('myPdfSign self-test failed — check PdfSignClass.clw is current.')
  END

This confirms the DER/ASN.1 reader and the dictionary scan are wired correctly before you depend on them in production.

Scope & honesty #

What myPdfSign answers — and what it does NOT myPdfSign surfaces the named signer identity (the certificate Subject, issuing CA and signing time) plus an integrity hint (CoversWholeFile). It does NOT cryptographically verify the RSA/ECDSA signature, and it does NOT validate the certificate trust chain — there is no hash check, no revocation lookup, and no CA-trust evaluation.

So the question it answers is precisely: “who is named as the signer, and was anything appended after signing?” Treat the identity as asserted by the document — it is cryptographically extractable from the embedded PKCS#7, but it is not crypto-verified. That is the right tool for triage, audit display, indexing and routing; it is not a replacement for a full signature-validation library when legal non-repudiation is on the line.

One subtlety worth knowing: SignTime is read from the PKCS#7 signed attributes (the authenticated signingTime), not from the dictionary's spoofable /M entry — so it reflects what the signer's tooling actually committed to at signing time.

How it works #

No crypto math — identity and integrity only. The reader is four steps:

  1. Find the signature dictionary. Scan the PDF for /ByteRange [ … ] and /Contents <hex> — the entries every detached PDF signature carries.
  2. Hex-decode /Contents. The hex string between the angle brackets decodes to the DER-encoded PKCS#7 SignedData blob (the reserved trailing zeros are ignored — DER is self-delimiting, so the true object length is trimmed).
  3. Walk the DER. A tiny ASN.1 tag/length reader descends ContentInfo → SignedData → certificates[0] → Certificate → tbsCertificate, then reads the Subject and Issuer RDNs by OID, and the signingTime from the authenticated attributes:
    FieldOID
    commonName (CN)2.5.4.3
    organizationName (O)2.5.4.10
    organizationalUnitName (OU)2.5.4.11
    emailAddress1.2.840.113549.1.9.1
    signingTime (signed attr)1.2.840.113549.1.9.5
  4. Check whole-file coverage. Confirm the /ByteRange two segments ([0, a) and [b, len-b)) together span the entire current file. If they do, CoversWholeFile = 1; if bytes were appended after signing, the second segment no longer reaches the end and CoversWholeFile = 0 — a tamper / incremental-update hint.

Architecture #

templates/myPdfSign/ ├─ PdfSignClass.inc the class: result fields, working buffers, method prototypes ├─ PdfSignClass.clw the engine (dictionary scan, hex decode, DER/ASN.1 walk) └─ myPdfSign.tpl one global extension (adds the shared object) designer/PdfSignCore/ the .NET golden-fixture oracle (validation)

The class is self-contained the same way myCompress's codec is: a CLASS,TYPE,MODULE,LINK whose .clw opens with a bare MEMBER and a module-level MAP (so the BUILTINS.CLW intrinsics and the Win32 file calls resolve). All working state — the loaded PDF bytes, the decoded DER blob and its trimmed length, and every result field — lives in class members, so the single global instance is reusable and has no global side-effects.

The internals split cleanly into a PDF-structure layer and a DER/ASN.1 layer. FindContents locates the /Contents <hex>, HexToDer decodes it, GetParen / GetName read the dictionary's /Key(…) and /Key/value entries, and ByteRangeCovers does the whole-file check. Underneath, DerHdr / DerNext / DerInto are the tag/length primitives that ParseSigner, GetAttr and GetSignTime use to walk to the certificate's RDNs and the signed signingTime.

The .NET oracle #

Clarion cannot be run from CI, so — as with CompressCore and QrCodeCore — a small .NET tool, designer/PdfSignCore/, is the independent reference. .NET owns the trusted building blocks (X.509 certificate creation and PKCS#7 / CMS via SignedCms), so the tool both manufactures the fixtures and publishes the ground truth the Clarion side must reproduce.

  • Manufacture — for each case it makes a real root CA, then a leaf certificate signed by that CA (so the issuer DN is genuinely distinct from the subject DN), assembles a minimal one-page PDF with an AcroForm signature field, and writes a detached PKCS#7 signature over a proper /ByteRange with the signingTime in the signed attributes.
  • Ground truth — it re-parses each finished PDF with SignedCms and writes a caseN.expected.txt key=value identity block that the PDF must yield.
  • Verify — the Clarion app runs Report() over each fixture into caseN.actual.txt, and the tool compares it to the expected truth byte-for-byte.
# manufacture caseN.pdf + caseN.expected.txt + manifest
dotnet run --project designer/PdfSignCore -- vectors ./vectors
# …have the Clarion app Read each caseN.pdf and write Report() to caseN.actual.txt…
dotnet run --project designer/PdfSignCore -- verify  ./vectors    # compares actual vs expected

Three fixture cases all pass: case1 (Alice Anderson, Redding Assessments / Redding Root CA), case2 (Bob Builder, Acme Construction Ltd / Acme Internal CA), and case3 (Carol O'Neil, Health Trust) — which is deliberately tampered by appending bytes after %%EOF, and which the Clarion class correctly reports as CoversWholeFile=0. The selfcheck command manufactures and re-parses every case in-memory as a fast sanity pass.

Class reference #

Everything myPdfSign does lives in one class, PdfSignClass. This chapter is the full anatomy: how it's declared, its lifecycle, and every public property and method.

Declaration

PdfSignClass CLASS,TYPE,MODULE('PdfSignClass.CLW'),LINK('PdfSignClass.CLW')
  • TYPE — a type, so you declare your own named instance (the template declares one global PdfSig). You could declare more if you needed independent readers.
  • MODULE/LINK — the engine compiles into its own object and links in automatically.
  • The .clw carries a bare MEMBER + module-level MAP (folds in BUILTINS.CLW and the file calls). No external state — safe in multi-DLL apps and across threads.

Lifecycle

Construct runs automatically and clears the result fields. Each call to ReadFile / Read resets the results first (via the internal Reset) and then refills them. ReadFile owns the bytes it loads; Destruct frees that buffer and the decoded DER. Nothing else needs cleanup.

Methods #

The four public methods. ReadFile / Read / SelfTest return a BYTE (1 = ok); Report returns a STRING.

PrototypeParametersReturns / effect
ReadFile(STRING pPath),BYTE pPath — path to the PDF on disk. Loads the file (owning its own copy of the bytes) and parses it. Returns 1 when it parsed — then check Signed for whether a signature was present, and ErrText on any failure.
Read(*STRING pPdf, LONG pLen),BYTE pPdf — a PDF already in memory. pLen — its real byte count. Parses the in-memory buffer in place (does not take ownership). Returns 1 when parsed. Fills the same result properties as ReadFile.
Report(),STRING — Returns the key=value identity block matching the PdfSignCore oracle's expected output — for logging, diffing, or display.
SelfTest(),BYTE — Parses a tiny built-in signed PDF and checks the extracted identity. Returns 1 on pass — a one-line startup smoke test.

Troubleshooting #

SymptomCause / fix
ReadFile returns 1 but Signed = 0The PDF has no signature dictionary — it is genuinely unsigned. Check SigCount.
Read mis-parses near the end of the bufferYou passed the wrong pLen (or a padded STRING). Always pass the exact byte count you loaded.
CoversWholeFile = 0 on a file you trustBytes exist after the signed /ByteRange — an incremental update or appended content after signing. That is exactly what this flag reports.
Empty SubjectEmail / SubjectOUThe certificate simply did not carry that RDN. Only OIDs present in the Subject are filled; the e-mail must be an explicit emailAddress RDN, not only a SAN extension.
"Unresolved external" on a built-inThe .clw must open with a bare MEMBER and a module-level MAP so BUILTINS.CLW resolves. Don't remove them.
Template won't registerRe-save the class files as ANSI, not UTF-8.
I need to know the signature is cryptographically validOut of scope — see Scope & honesty. myPdfSign extracts the named identity and the whole-file hint; it does not verify the RSA/ECDSA signature or the trust chain.
