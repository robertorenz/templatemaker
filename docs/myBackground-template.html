myBackground

A self-contained, global Clarion 12 ABC extension that gives every window a default background — a solid colour or an image — and lets the user set a personal background per window with Ctrl+Shift+B, persisted to an INI file. This is the programmer's reference: prompts, generated code, embed points, the helper API, and the runtime properties it relies on.

myBackground · version 1.00 · FAMILY('ABC')

Overview ↑ top

One APPLICATION-scope extension, added once at the global level, with no per-procedure setup. It does three things:

  1. Global default — every window opens with a default background (a solid colour and/or an image) you configure on the extension's prompts.
  2. Per-window override — pressing Ctrl+Shift+B on any window pops a small chooser to set a personal colour or image for that window, or revert it to the default.
  3. Persistence — each window's personal choice is written to its own section of an INI file and re-applied automatically the next time the window opens. A personal background overrides the global default.

It is fully self-contained: two helper procedures are generated straight into the program module (prototypes in the global MAP, bodies in %ProgramProcedures). There are no external .inc/.clw files to ship.

Precedence For any window: a stored personal background wins; otherwise the global default image wins; otherwise the global default colour; otherwise the window keeps its standard look.

Install & register ↑ top

  1. Copy templates/myBackground/myBackground.tpl into your app's template/source path (or register it in place).
  2. In the IDE: Setup ▸ Template Registry ▸ Register and pick myBackground.tpl.
  3. Open your app, go to Global ▸ Extensions, and add myBackground - per-window background color / image.
  4. On the General tab set your defaults and INI name (all optional — see below).
  5. Generate, compile, and run.
Scope Add it once, at the application/global level. Because it injects into WindowManager.TakeWindowEvent, it automatically reaches every windowed procedure in the app — there is nothing to add to individual procedures.

Runtime behaviour ↑ top

On EVENT:OpenWindow the template calls myBackApply() for the window and (if the hotkey is enabled) arms Ctrl+Shift+B. myBackApply resolves the precedence:

! pseudocode of the decision in myBackApply
IF window has a stored personal setting   ! INI Mode = 'C' or 'I'
   apply that colour / image
ELSIF a default image is configured        ! %mbgDefImage not blank
   apply the default image (stretched)
ELSIF a default colour is configured       ! %mbgDefColor <> COLOR:None
   apply the default colour
ELSE
   clear to the standard window look

A colour is set with 0{PROP:Color} (the current window). An image is set with 0{PROP:WallPaper} after turning PROP:Tiled and PROP:Centered off, so the picture stretches to fill the whole client area. Applying a colour first clears any image, and vice-versa, so the two modes never fight.

Why a draw at OpenWindow works The handler is injected at PRIORITY(2000) — a self-contained CASE EVENT() above the ABC framework's own block (which opens at 2500) — and it never RETURNs, so normal event handling continues untouched. This is the same proven pattern used by the sibling myFontChanger and myPixel templates.

The Ctrl+Shift+B chooser ↑ top

When the user presses the hotkey, EVENT:AlertKey fires and the template calls myBackChoose(), which opens a small modal window:

Window Background
  Set the background for this window:
  [ Background Color… ]
  [ Background Image… ]
  [ Use Default ]  [ Close ]
ButtonAction
Background Color…Opens COLORDIALOG. On OK, saves Mode=C + the chosen colour, clears the image entry, and applies it.
Background Image…Opens FILEDIALOG (BMP/JPG/GIF/PNG/ICO/WMF). On OK, saves Mode=I + the path, clears the colour entry, and applies it stretched.
Use DefaultDeletes this window's whole INI section and re-applies the global default.
CloseDismisses the chooser with no change.
Targeting the right window The chooser opens its own window, so it CLOSEs that window before re-applying the background. After the close, the caller's window is again the top window on the thread, so 0{PROP:…} inside myBackApply correctly targets the window the user was on — not the chooser.

INI storage ↑ top

Each window is keyed by its procedure name (unique within an app), so every window gets its own [section]. Three entries per section:

EntryValuesMeaning
ModeC / I / (absent) Colour, Image, or no personal setting (use the global default).
ColorCOLORREF LONGThe solid colour when Mode=C.
Imagefile pathThe image when Mode=I.

Example myBackground.INI:

[BrowseClients]
Mode  = C
Color = 16771818          ! a pale blue COLORREF
Image =

[UpdateOrder]
Mode  = I
Color = -1
Image = .\art\linen.jpg

The default path .\myBackground.INI is relative to the program's working directory. Point %mbgIni at a per-user writable location (e.g. under %APPDATA%) if the install folder is read-only.

Embed points & generated code ↑ top

The template injects at three embed points, all guarded by WHERE(%mbgDisable=0):

EmbedWhat it emits
%GlobalMapShort-form prototypes for myBackApply and myBackChoose (short form survives the MAP's auto-indent).
%ProgramProceduresThe two helper bodies, long-form, at column 1 (this region is not auto-indented).
%WindowManagerMethodCodeSection,'TakeWindowEvent','(),BYTE'
PRIORITY(2000), WHERE(… AND %Window)		 The per-window handler — apply background + arm the hotkey on open, pop the chooser on the key.

The generated handler (with the hotkey enabled) looks like:

  CASE EVENT()
  OF EVENT:OpenWindow
    myBackApply('BrowseClients', '.\myBackground.INI', -1, '')
    ALERT(CtrlShiftB)
  OF EVENT:AlertKey
    IF KEYCODE() = CtrlShiftB
      myBackChoose('BrowseClients', '.\myBackground.INI', -1, '')
    END
  END
Hotkey off When %mbgHotKey is 0, the ALERT line and the whole OF EVENT:AlertKey arm are #IF'd out — the CASE stays well-formed with just the OpenWindow arm.

Helper API ↑ top

Both helpers are generated into the program module and act on the current window (handle 0). You can call them yourself from embed code if you want to drive the background programmatically.

myBackApply

myBackApply(STRING pKey, STRING pIni, LONG pDefColor, STRING pDefImage)
ParamMeaning
pKeyINI section name (the window's procedure name).
pIniINI file path.
pDefColorGlobal default colour, or COLOR:None (-1) for none.
pDefImageGlobal default image path, or blank for none.

Reads the section, resolves the precedence, and applies the result to window 0. Safe to call any time the target window is the current window (e.g. EVENT:OpenWindow).

myBackChoose

myBackChoose(STRING pKey, STRING pIni, LONG pDefColor, STRING pDefImage)

Same parameters. Opens the chooser, writes the user's choice to the INI, then re-applies it to the caller window via myBackApply. Call it from a button or menu if you want a "Change background…" command somewhere other than the hotkey.

Reapply after a programmatic change If your own code edits the INI directly, call myBackApply(...) on the live window to repaint it without reopening.

Customising ↑ top

  • Different hotkey — change both ALERT(CtrlShiftB) and the IF KEYCODE() = CtrlShiftB test in the %WindowManagerMethodCodeSection block to another keycode equate (see KEYCODES.CLW).
  • Tiled or centered images — in myBackApply's image branches set 0{PROP:Tiled}=1 (or PROP:Centered=1) instead of 0.
  • Per-window opt-out — change the #AT guard to also test a per-procedure symbol, or add a #EXTENSION(...),PROCEDURE disable flag, if some windows should be skipped.
  • Image filter — edit the extension string in the FILEDIALOG call inside myBackChoose to narrow or widen the accepted formats.

Multi-DLL note ↑ top

The helper bodies are emitted into %ProgramProcedures (the program/EXE module), matching the sibling templates (myFontChanger, myFuncs). This targets single-EXE apps. For a multi-DLL app where windowed procedures live in DLLs, move the helper bodies to a shared module and add the prototypes (and exports) to the DLLs that call them — the same retrofit any self-contained helper template needs.

Troubleshooting ↑ top

Template won't registerA literal % in an emitted line, or an unbalanced block. This template is clean; if you edit it, escape any literal % as %% and keep #SHEET/#TAB/#BOXED/#AT/#IF pairs matched.
Default image ignoredThe path must be reachable at run time. Relative paths resolve from the working directory; use a full path or ship the art folder.
Hotkey does nothing"Enable Ctrl+Shift+B picker" is off, or another extension consumes the key. The alert is armed on EVENT:OpenWindow per window.

Setting doesn't persistThe INI path isn't writable. Point %mbgIni at a per-user writable location.

Verify checklist ↑ top

Claude can't run AppGen — after generating, confirm:

  1. The .tpl registers without error.
  2. A window with a default colour/image set shows it on open.
  3. Ctrl+Shift+B opens the chooser; Colour and Image both apply at once.
  4. The choice survives closing and reopening the window (check the INI section was written).
  5. Use Default removes the personal setting and reverts to the global default.
