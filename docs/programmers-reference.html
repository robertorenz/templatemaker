Programmer's Reference

How the Clarion Template Designer is built — the parse/model/write pipeline, the relocation engine that keeps edits surgical, and a working reference to the slice of the Clarion template language the designer understands.

Programmer's Reference · version 2.8

Overview ↑ top

The designer is a .NET 9 / WPF application. Its core is a small, lossless pipeline: parse a template into a model, let the UI mutate that model, then write only what changed back into the original text. The guiding rule is preserve every byte you didn't touch.

// the whole pipeline, conceptually
TplParser.Parse(path)   → TplDocument (Files, Components, Tabs/Elements)
        ↓  user edits flip Dirty / Inserted / Deleted / Moved flags
TplWriter.Save(doc)     → rewrites only changed lines, per file

The model ↑ top

Four record-like classes in Tpl.cs hold the parsed template:

TypeKey members
TplElementOne control or container. Kind, Title, Symbol, PromptType; entry position X,Y,W,H + HasX/Y/W/H; label position PX,PY + HasPromptAt (the PROMPTAT for a prompt); Section (the box is a #BOXED,SECTION); style FontName,FontSize,FontColor, FontStyle,Bold with computed Italic/Underline/Strikeout from the style flag bits; line span LineIndex,EndLineIndex; edit flags Dirty, Inserted, Deleted, Moved, FontDirty; relocation MoveAnchorLine; tree Parent, Children; computed layout LX/LY/LW/LH (entry) and PLX/PLY/PLW/PLH (label). Clone() deep-copies for undo.
TplFilePath, Newline, Lines[], Included, Dirty. Dirty flags raw-text edits (e.g. a symbol rename) so the writer saves the file even with no element changes.
TplComponentA #EXTENSION/#CONTROL/#PROCEDURE/#CODE/#GROUP/…: Kind, Name, FileIndex, StartLine, EndLine, Tabs; HasSheet.
TplDocumentPath, Files, Components, Newline — the whole loaded set.

IsContainer = Tab / Boxed / Button / Enable. IsPositionable = Prompt / Display / Image / Boxed.

Parser ↑ top

  • TplParser.Parse(path) — loads the main file and follows #INCLUDE('…') recursively (cycle-guarded; #! comment lines ignored).
  • TplParser.ParseText(text, path) — parses an in-memory single file (no includes); used to map the live/pending editor text back onto the model.
  • ParseComponents walks the directive structure; ParseSheet builds the #SHEET/#TAB/#BOXED/#PROMPT/#DISPLAY/#IMAGE tree with line indices.

Writer ↑ top

  • TplWriter.Save(doc) — writes each file whose TplFile.Dirty is set, or that contains an element flagged Dirty/Inserted/Deleted/Moved/FontDirty.
  • PreviewFile(doc, fileIndex) — the text a file would have, without writing (drives Live preview).
  • BuildLines(file, docTabs) — the heart of the writer.

BuildLines clones file.Lines as its base, so any direct edit to that array (such as a rename sweep) flows straight through. It then:

  1. builds a drop set — deleted ranges, and the full range of any moved container;
  2. does an in-place AT/PROP rewrite for edited-but-not-relocated elements;
  3. runs the emit loop, re-inserting added/moved units at their anchor.

Relocation model ↑ top

Re-ordering, reparenting and insert-at-position all share one mechanism:

  • An element being relocated is flagged Moved with a MoveAnchorLine — the line of the next real sibling it should be emitted before (or its owner's end line).
  • The drop set removes the element's original lines; the emit loop re-emits it at the anchor.
  • A moved container is emitted as its verbatim block (only the open line gets AT/PROP rewrites), so its inner formatting survives intact.
  • Owners that can hold children — Tab, Boxed, Enable and Button — are all honoured by the emit loop, so a control whose parent is any of them is never dropped without being re-emitted.
Invariant Every relocation must be net-zero on line count and keep the block balanced (e.g. #BOXED…#ENDBOXED). New writer paths are validated by re-parsing the output and checking structure before shipping.

Symbol rename ↑ top

Renaming a %Symbol keeps the prompt and all its references joined by sweeping raw text:

var rx = new Regex(Regex.Escape(oldSym) + @"(?![A-Za-z0-9_])"); // word boundary
foreach (var f in doc.Files) {
    // replace every hit in f.Lines, set f.Dirty if touched
}
// then update Symbol on every matching model element to stay joined
  • Matching is word-boundary so %Cust never touches %CustName.
  • Renaming into an already-existing symbol is blocked (it would silently merge two fields).
  • Touched files are flagged TplFile.Dirty so Save writes them.
  • The full raw text is captured in the undo snapshot, so a rename is undoable.

The Properties pad's Uses list comes from the same scan (FindUsages) over every file; clicking an entry switches part/file as needed and scrolls the editor to the line.

Undo ↑ top

A Snapshot deep-clones the component tabs, the guides, and — since v2.3 — each file's raw Lines + Dirty flag. Drag/resize gestures capture once at the start and commit only if something actually changed. Structural saves re-parse from disk to re-sync line indices and clear the undo stack.

Clarion directives ↑ top

The designer reads the prompt-UI slice of the template language:

DirectiveMeaning
#TEMPLATE / #EXTENSION / #CONTROL / #PROCEDURE / #CODE / #GROUP / #UTILITYComponent kinds.
#SHEET / #TABThe tabbed prompt container and its pages.
#BOXED … #ENDBOXEDA group box holding child controls.
#PROMPT / #DISPLAYAn input field (bound to a %Symbol) / a static label.
#IMAGEA picture control (renders the PNG on the canvas).
#BUTTON / #ENABLEA button / a conditional-enable container.
#INCLUDE('x.tpw')Pulls another template file into the set.

AT() & PROP() ↑ top

AT(x, y, w, h) positions a control's entry in dialog units; a prompt's label is positioned separately by PROMPTAT(x, y). Any omitted slot is left to Clarion's auto-layout — the designer tracks HasX/HasY/HasW/HasH (and HasPromptAt) so it never invents a coordinate the author didn't write. ApplyAt rewrites AT and ApplyPromptAt the PROMPTAT; \bAT\( deliberately doesn't match inside PROMPTAT(.

Inside a #BOXED, child coordinates are tab-absolute unless the box has SECTION (then they're box-relative). The designer's model is box-relative, so the writer auto-adds SECTION to any box holding positioned children.

Style is carried as PROP attributes, rewritten on style edits:

PROPCarries
PROP:FontNameFont face name — the string element (PROP:Font is the 4-value array, so the face must use PROP:FontName).
PROP:FontSizePoint size.
PROP:FontStyleWeight (400/700) plus flag bits: italic 0x1000, underline 0x2000, strikeout 0x4000.
PROP:FontColorText colour as a COLORREF.
Prompt-sheet face is the IDE font Clarion renders prompt sheets in the AppGen Dialogs font (ClarionProperties.xml ▸ AppGen Dialogs, e.g. Segoe UI 9), which the designer reads on startup to drive rendering and the dialog-unit calibration. Per-control PROP:FontName/PROP:FontSize aren't honoured there, so the UI exposes only bold/italic/underline/colour and shows the face read-only.

Colours & keycodes ↑ top

COLORREF

Clarion colours are Win32 COLORREF values — R | (G<<8) | (B<<16) — written in hex as 0{value:X}H. The designer converts to/from WPF colours when you pick one, and removes the PROP entirely when you clear the colour.

Keycodes

A keycode is the Windows virtual-key low byte plus modifier bits: Shift 0x100, Ctrl 0x200, Alt 0x400. The designer decodes these to a friendly name (e.g. Ctrl+Shift+P) in the Properties pad.

Prompt types ↑ top

ClassifyPrompt recognises the prompt's picture/type token and renders the right editor in the flow preview: text pictures (@s, @n), CHECK, OPTION/RADIO, SPIN, DROP/FROM lists, PROCEDURE, EXPR, KEYCODE, and the dialog “…” pickers.

Auto-built prompts Some prompt types make Clarion build a dropdown or “…” button next to the field and lay it out for you. The designer flags these and skips them in “Add AT to all”, because pinning an explicit position can hide that auto part.

Build & package ↑ top

# build the app
dotnet build designer/ClarionTplDesigner/ClarionTplDesigner.csproj -c Release

# portable single-file exe  → run/ClarionTemplateDesigner.exe
pwsh installer/build-portable.ps1

# full installer  → installer/Output/ClarionTemplateToolsSetup.exe
pwsh installer/build-installer.ps1

The version lives in two places that must stay in sync: <Version> in the .csproj and #define AppVersion in the .iss. After a designer change, rebuild the portable exe into run/.

a new relocation/insert paththe emit loop in BuildLines — then validate by re-parsing the output

Golden rule Any change to the writer must be proven net-zero on line count and structurally balanced by re-parsing its own output before it ships.
