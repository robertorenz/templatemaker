myQRDraw

An offline QR-code generator for Clarion windows. It encodes the QR symbol itself at run time and draws every module as a filled BOX into an IMAGE control — no internet, no curl, no temporary image files.

ABC & Clarion · versions 1–10 · ECC L/M/Q/H · self-contained

Overview #

myQRDraw turns any value — a URL, an order number, a vCard string — into a scannable QR code painted directly onto a window, entirely on the local machine.

It is the companion to myQR. Where myQR downloads a PNG from api.qrserver.com (and therefore needs a working internet connection), myQRDraw carries a complete QR encoder and draws the result with the same graphics primitives myPie uses for its chart. That makes it ideal for kiosks, point-of-sale, field laptops, air-gapped networks, and reports that must render with zero external dependencies.

Install & register #

  1. Copy templates/myQRDraw/myQRDraw.tpl into your template folder.
  2. In the IDE: Setup ▸ Template Registry ▸ Register, pick myQRDraw.tpl.
  3. The chain myQRDraw registers two extensions: the global helper and the per-window drawer.

Quick start #

  1. Open your application. On the global Extensions add “myQRDraw - Global Encoder + Helper” (once).
  2. On a window procedure, add an IMAGE control (not a Region — Clarion draws into Image controls). Make it roughly square and big enough (≥ ~60×60 dialog units for small codes).
  3. Add the procedure extension “myQRDraw - Draw a QR Code on this window”.
  4. On its General tab: pick the IMAGE control, type the Value (e.g. a URL), choose the ECC level, the Dark/Light colors and the quiet-zone width.
  5. Generate, compile, run — then scan the code with a phone.
It draws on open and on resize The code repaints on EVENT:OpenWindow and EVENT:Sized. Set the IMAGE to resize/anchor and the code follows the window, always re-centred and crisp.

Self-test #

Tick “Draw self-test (HELLO WORLD, ECC M)” on the General tab. The window then draws a fixed 21×21 version-1 symbol that encodes the literal text HELLO WORLD, ignoring the Value field.

Scanning it must read exactly HELLO WORLD. This is the fastest way to confirm the offline encoder works correctly on your machine — the very same matrix is pinned by an automated test (see Verification).

Changing it at run time #

Tick “Value is a variable” and name a variable in the Value field. Change the variable, then call the generated repaint routine:

MyWebVar = 'https://newsite.example/order/4471'
DO myQRDrawRepaint        ! re-encodes and redraws into the image

myQRDrawRepaint simply posts a private event so the redraw happens cleanly inside the window's event loop, after which the code is re-encoded from the current variable value.

Architecture #

The global extension emits three things, guarded by its Disable check:

  • %GlobalData — the encoder's working arrays, all QR:-prefixed and sized for the largest supported symbol (version 10 = 57×57): the module matrix, the function-module map, the GF(256) log/antilog tables, codeword buffers, and the capacity table.
  • %GlobalMap — short-form prototypes for every encoder procedure.
  • %ProgramProcedures — the procedure bodies: the encoder and the public QRDraw() helper.

The procedure extension adds a tiny per-window slice: a private repaint event, a value buffer, a myQRDrawRepaint routine, and a handler at the top of TakeWindowEvent that calls QRDraw() on open/resize — exactly the structure myPie uses.

Multi-DLL note Like myPie, the bodies live in %ProgramProcedures (the program module). For a multi-DLL application, move the encoder to the root/shared DLL and export the prototypes.

The encoder #

The Clarion encoder is a faithful, line-for-line port of a C# reference implementation (designer/QrCodeCore) that is validated by decoding its output with ZXing. It implements the full byte-mode pipeline:

  • Data codewords — mode indicator, character count, the UTF-8/byte payload, terminator and pad bytes (0xEC / 0x11).
  • Reed–Solomon error correction over GF(256) with primitive polynomial 0x11D, the generator built per block.
  • Block interleaving for multi-block versions.
  • Function patterns — finders, separators, timing, alignment, the dark module, and (for versions ≥ 7) the 18-bit version information.
  • Codeword placement — the zig-zag walk that skips function modules and the timing column.
  • Masking — all eight mask patterns are tried; the one with the lowest penalty (the four standard rules) wins.
  • Format information — the 15-bit BCH format string (ECC + mask) placed in both copies.
Two Clarion-specific care points Clarion rounds on assignment to an integer, so every truncating divide uses INT(); and modulus is done with a small QRMod() helper (a - INT(a/b)*b) so no literal % ever appears in the emitted source.

Drawing #

Once the matrix is built, QRDraw() paints it just like myPie paints its pie:

  1. SETTARGET(,image) to draw into the IMAGE control, then GETPOSITION to read the control's X,Y (drawing is window-relative — GitHub issue #5).
  2. BLANK to wipe prior graphics (so a resize leaves no artifacts).
  3. A single light BOX fills the whole control (the quiet zone is light too).
  4. Each dark module becomes one filled BOX of cell × cell dialog units.

The cell size is computed to fit the control — cell = INT(min(width,height) / (modules + 2·quiet)) — and the whole symbol is centred. If the control is too small the cell floors at 1 dialog unit.

Verification & tests #

The encoder's correctness is anchored by an automated test project, designer/QrCodeCore.Tests:

  • Round-trip — every encode is decoded back with ZXing across all versions and ECC levels.
  • Reed–Solomon — pinned to the ISO/IEC 18004 worked example.
  • Golden matrix — the exact 21×21 module grid for HELLO WORLD at ECC M is locked. That is the same symbol the Self-test draws, so a phone scan that reads HELLO WORLD proves the Clarion port reproduces the tested encoder module-for-module.

Run them with dotnet test designer/QrCodeCore.Tests.

QRDraw() API #

QRDraw(SIGNED pImageFeq, *CSTRING pValue, LONG pEcc, |
       LONG pDark, LONG pLight, LONG pQuiet)
ParameterMeaning
pImageFeqThe IMAGE control's field-equate to draw into.
pValueThe text/bytes to encode.
pEccError correction: 1=L, 2=M, 3=Q, 4=H.
pDark / pLightModule / background colors.
pQuietQuiet-zone width in modules.

The helper encodes and draws in one call and is safe to call repeatedly. If the value is too large for versions 1–10 at the chosen ECC, it returns without drawing, leaving the control as-is.

Capacity & limits #

  • Versions 1–10 are supported (21×21 up to 57×57 modules).
  • Byte mode is used for everything — URLs, numbers, and text all work; the version is chosen automatically as the smallest that fits.
  • Approximate byte capacity at version 10: L ≈ 271, M ≈ 213, Q ≈ 151, H ≈ 119 bytes.
  • Higher ECC survives more damage/dirt but stores fewer bytes. M is a good default.
Need more than ~271 bytes? Shorten the payload (e.g. a short URL that redirects) or raise the image so a denser code still scans. Versions above 10 are not currently generated.
