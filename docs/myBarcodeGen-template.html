myBarcodeGen — Developer's Manual

A Clarion template set that encodes and draws nine barcode symbologies directly into an IMAGE control with BOX primitives — offline, no internet, no curl, no temp files. Five self-contained encoder classes, ported from a ZXing-validated C# reference.

Code 39 · Code 128 · ITF · EAN-13 · UPC-A · QR · Data Matrix · PDF417 · Aztec

Overview #

myBarcodeGen turns any value — a part number, a URL, a GTIN, a shipping payload — into a scannable barcode painted onto a window or a report, entirely on the local machine.

It generalises the myQRDraw approach (encode at run time, draw with BOXes) to a whole family of symbologies. Each encoder is a faithful Clarion port of a C# reference (designer/BarcodeCore) that is validated by decoding its output with ZXing — 42 automated round-trip tests across all nine codes.

The shape of the template One global extension carries the encoders (add once per app); a window extension and a report extension each draw a barcode of the type you pick. The QR type reuses the same QRCodeClass as the myQRDraw template.

Install & the class files #

  1. Copy templates/myBarcodeGen/myBarcodeGen.tpl into your template folder and register it: Setup ▸ Template Registry ▸ Register.
  2. Copy the five encoder classes — ten .inc/.clw files — to a folder on the Clarion redirection path (your app folder, or \clarion12\libsrc\win).
templates\myBarcodeGen\
  myBarcodeGen.tpl    ← register this
  BarcodeClass.inc / .clw   ← 1D codes (Code 39/128, ITF, EAN-13, UPC-A)
  DataMatrixClass.inc / .clw ← Data Matrix
  Pdf417Class.inc / .clw   ← PDF417
  AztecClass.inc / .clw    ← Aztec
templates\myQRDraw\
  QRCodeClass.inc / .clw   ← QR (shared with myQRDraw)
Two hard requirements for the class files
  • Store every .inc/.clw in ANSI (not UTF-8). A BOM or any multi-byte character will break the Clarion compiler.
  • They must be on the redirection path so INCLUDE and the class LINK attribute can find them — the .clw modules compile automatically (no project edit).
Only need some types? The global extension includes all five classes. If you only ever use, say, the 1D codes, you can omit the classes you don't reference — but you must also remove their INCLUDE/instance lines from the generated global module. The simplest path is to ship all ten files.

Quick start #

  1. On the global Extensions of your app, add “myBarcodeGen - Global Encoders” (once).
  2. On a window procedure, add an IMAGE control (not a Region). Make it wide for 1D barcodes, or square-ish for the 2D codes.
  3. Add the procedure extension “myBarcodeGen - Draw a barcode on this window”.
  4. Pick the IMAGE control and the Barcode type, type a Value, choose colors and the quiet zone.
  5. Generate, compile, run — then scan with a phone or scanner.
Self-test Tick “Draw self-test sample for the chosen type” to draw a fixed, known-good value for whatever symbology you selected (e.g. HELLO WORLD for QR, 5901234123457 for EAN-13). Scanning it confirms the encoder works on your machine.

Barcode types #

The Barcode type prompt is a drop-list; its stored value (1–9) is the type code used at generation time to wire up the right class. The codes are also available as equates from BarcodeClass.inc, handy if you call the classes directly:

BC:Code39=1  BC:Code128=2  BC:ITF=3  BC:EAN13=4  BC:UPCA=5  BC:QR=6  BC:DataMatrix=7  BC:PDF417=8  BC:Aztec=9

The template dispatches at generation time (not run time): because the type is chosen in the designer, only the call for that one symbology is generated. Types 1–5 go to BarcodeObj (BarcodeClass); 6→QRCodeObj; 7→DataMatrixObj; 8→Pdf417Obj; 9→AztecObj.

Window extension #

myBarcodeGen - Draw a barcode on this window draws into the chosen IMAGE control and repaints on EVENT:OpenWindow and EVENT:Sized — set the IMAGE to resize/anchor and the barcode follows the window. It also generates a repaint routine:

loc:Value = 'https://example.com/order/4471'
DO myBarcodeGenRepaint   ! re-encode + redraw from the current value

Behind the scenes the handler builds myBC:Value (a CSTRING(512)) from the Value prompt — literal text, or a Clarion variable/expression if you tick “Value is a variable” — then calls the appropriate class.

Report extension #

myBarcodeGen - Draw a barcode on this REPORT draws one barcode per record into an IMAGE control in a band. Reports render through the print engine, not a window event loop, so the drawing happens in the Before-Print-Detail embed with SETTARGET(Report):

  • Put an IMAGE control in the detail band (or any band that prints per record) and give it a USE / field-equate.
  • The control picker lists the report's controls (it uses FROM(%ReportControl,…), not a window CONTROL prompt).
  • Point the Value at a per-record field (e.g. ORD:Barcode) for a unique code each row.
Verify placement on a printout Report band-drawing is timing-sensitive. Use the self-test and check a printout/preview; if a code lands in the wrong place, confirm the IMAGE is in the detail band.

Run-time / calling the classes directly #

You don't have to use the prompts — the encoder objects are global, so you can draw from any code. The objects (declared by the global extension) are BarcodeObj, QRCodeObj, DataMatrixObj, Pdf417Obj, AztecObj.

! 1D — type, value, dark, light, quiet, show-text
BarcodeObj.Draw(?Image, BC:Code128, 'ABC-123', COLOR:Black, COLOR:White, 10, 1)

! QR — value, ecc(1..4), dark, light, quiet
QRCodeObj.Draw(?Image, 'https://site/x', 2, COLOR:Black, COLOR:White, 4)

! Data Matrix / PDF417 / Aztec — value, dark, light, quiet
DataMatrixObj.Draw(?Image, 'PART-9912', COLOR:Black, COLOR:White, 2)
Pdf417Obj.Draw(?Image, loc:Payload, COLOR:Black, COLOR:White, 2)
AztecObj.Draw(?Image, loc:Ticket, COLOR:Black, COLOR:White, 2)

Each Draw targets the IMAGE control, clears it (BLANK) and paints in one call. If the value can't be encoded (too large, or invalid characters for the symbology) it returns without drawing, leaving the control unchanged.

1D linear codes #

  • Code 39 — characters 0–9 A–Z and - . space $ / + %. Input is upper-cased; the * start/stop is added automatically. Self-checking (no check digit).
  • Code 128 — full ASCII 32–126. The encoder auto-uses Code C for all-digit, even-length data (compact), otherwise Code B. The mod-103 checksum is computed for you.
  • Interleaved 2 of 5 — digits only; an odd count gets a leading 0 (ITF needs an even number of digits). No check digit is added.
  • EAN-13 — 12 digits (the check digit is computed and appended) or 13 (used as-is). 95 modules.
  • UPC-A — 11 digits (check appended) or 12. Encoded as EAN-13 with a leading 0; the human-readable line shows the 12-digit UPC.
Human-readable text 1D codes can draw the value as a line of text beneath the bars (the “Show human-readable text” prompt). 2D codes ignore that option.

2D codes #

  • QR Code — byte mode, versions 1–10, ECC L/M/Q/H (the “QR error correction” prompt), automatic version + mask. See the myQRDraw manual for full detail.
  • Data Matrix — ECC200, ASCII encodation, square symbols 10×10…26×26 (single data region), chosen automatically by data size.
  • PDF417 — Byte compaction, automatic columns/rows and error-correction level. A stacked (multi-row) symbol; drawn with each row taller than a module.
  • Aztec — binary encoding, compact and full symbols, automatic layers and word size; the central bullseye is the finder (no quiet zone strictly required, but a small one helps scanners).

Trailing spaces are trimmed Every class CLIPs the value, so a space-padded fixed-length STRING field doesn't inflate the symbol. If a value is too large for its symbology (or has invalid characters for Code 39 / the digit-only codes), Build/Draw returns without drawing.

Architecture #

Two layers, mirroring myQRDraw:

  • C# reference — designer/BarcodeCore/ (+ BarcodeCore.Tests): the encoders written first and validated by decoding every output with ZXing.Net (42 round-trip tests). This is the oracle the Clarion ports are checked against; some tables (e.g. PDF417's 3×929 pattern table) are generated from it.
  • Clarion classes — five self-contained classes (.inc declaration + .clw methods), each declared CLASS,TYPE,MODULE('X.CLW'),LINK('X.CLW') so the .clw compiles into the build automatically. The template emits only an INCLUDE per class plus one global instance each, and calls into them.

The global extension wires everything at %AfterGlobalIncludes (the INCLUDEs) and %GlobalData (the instances). Each class builds a module pattern (1D = a bar array; 2D = a module matrix) then paints it with BOXes.

The classes & API #

Every class exposes a one-call Draw() (window use: set target, clear, paint), plus a Build()/Paint() pair used on reports (build once, paint into the already-set report target).

BarcodeClass — 1D (types 1–5), instance BarcodeObj

Draw(SIGNED pImageFeq, LONG pType, *CSTRING pValue, LONG pDark, LONG pLight, LONG pQuiet, LONG pShowText)
Build(LONG pType, *CSTRING pValue) ,BYTE   ! 1 ok / 0 invalid
PaintBars(SIGNED pImageFeq, LONG pDark, LONG pLight, LONG pQuiet, LONG pShowText)

pType is a BC: code (1–5). pShowText = 1 draws the human-readable line.

QRCodeClass — type 6, instance QRCodeObj

Draw(SIGNED pImageFeq, *CSTRING pValue, LONG pEcc, LONG pDark, LONG pLight, LONG pQuiet)
BuildMatrix(*CSTRING pValue, LONG pEcc) ,BYTE
Paint(SIGNED pImageFeq, LONG pDark, LONG pLight, LONG pQuiet)

pEcc: 1=L, 2=M, 3=Q, 4=H. (Note the build method is BuildMatrix, not Build.)

DataMatrixClass / Pdf417Class / AztecClass — types 7/8/9

Instances DataMatrixObj, Pdf417Obj, AztecObj — identical API:

Draw(SIGNED pImageFeq, *CSTRING pValue, LONG pDark, LONG pLight, LONG pQuiet)
Build(*CSTRING pValue) ,BYTE
Paint(SIGNED pImageFeq, LONG pDark, LONG pLight, LONG pQuiet)
Window vs report calls The window extension calls …Obj.Draw(…). The report extension calls IF …Obj.Build(value) then …Obj.Paint(…) with the report as the active target (QR uses BuildMatrix + Paint).

Drawing model #

  • 1D — the bar pattern is drawn as full-height vertical BOX bars, centred in the control with a quiet zone each side; contiguous dark modules merge into one BOX. Optional human-readable text is drawn with SHOW beneath the bars.
  • QR / Data Matrix / Aztec — a square module grid: a light background, then one BOX per dark module, scaled to fit and centred, with a quiet-zone border.
  • PDF417 — a stacked matrix (rows × columns of modules). Each barcode row is drawn taller than a module so the symbol has the right aspect ratio.

All drawing targets the IMAGE control via SETTARGET and is window/page-relative (GETPOSITION supplies the control's position and size). On a window, Draw also issues BLANK first so a resize leaves no artifacts.

Sizing Give 1D codes a wide IMAGE; give 2D codes a roughly square one. The module/bar size is an integer number of dialog units, so a larger control yields a larger, crisper code; the symbol is always centred.

Multi-DLL #

The encoder instances are global data, so the global extension declares them multi-DLL aware using ABC's standard symbols (no extra prompts): defined in a single-EXE or the root DLL, declared EXTERNAL,DLL(dll_mode) in other DLLs/EXEs, and exported from the root DLL.

#IF(%DefaultExternal = 'None External')
BarcodeObj BarcodeClass            ! single-EXE or root DLL: defined here
#ELSE
BarcodeObj BarcodeClass,EXTERNAL,DLL(dll_mode)   ! other DLL/EXE: imported
#ENDIF

Because the class methods are non-VIRTUAL and each class carries an unconditional LINK, every target links its own copy of the method code — only the shared instances are exported, so there's no method export-list to maintain.

Reed–Solomon fields #

The 2D codes use error correction over four different Galois fields. The encoders build the field tables (or compute the generator polynomial) at run time — worth knowing if you read the source:

QRGF(256)0x11D0
Data MatrixGF(256)0x12D1
PDF417GF(929) prime fieldmod 929 (primitive 3)1
AztecGF(2^n), n=6/8/10/120x43 / 0x12D / 0x409 / 0x10691

Testing & verification #

  • Automated (dev) — dotnet test designer/BarcodeCore.Tests runs 42 round-trip tests: every encoder's output is decoded back with ZXing and must equal the input (plus check-digit and structural assertions). That C# reference is the proven oracle the Clarion ports mirror.
  • On your machine — register the template, tick self-test for each type, generate/compile, and scan the drawn code. The 1D self-tests use valid sample values (e.g. a valid EAN-13); the 2D ones use short text.
The C# encoders are the source of truth The Clarion classes are ports; they are not compiled by the doc/test tooling here. If a generated app won't compile, the error is almost always a missing/UTF-8 class file or a stale copy — re-copy the ANSI .inc/.clw to the redirection path.

Troubleshooting #

SymptomCause / fix
“Unknown function label LEN / BOX / SETTARGET” compiling a classThe .clw needs its module-level MAP (it has one) — if you edited it, keep the MEMBER + MAP/END header that pulls in BUILTINS.CLW.
A < in a string won't compileClarion treats < as an <nnn> escape; literal < must be <<. The shipped tables avoid < entirely — don't hand-edit the packed PDF417 table.
Nothing drawsControl must be an IMAGE. Check the value isn't too large/invalid for the type (then Build returns 0). For reports, the IMAGE must be in the printed band.
Won't scanUse a larger/correct-aspect IMAGE (wide for 1D, square for 2D), keep a quiet zone, and use strong dark/light contrast (black on white).
1D barcode is huge/dense for short textA space-padded field is being encoded — values are CLIP'd, but make sure you're passing the trimmed field (or a CSTRING).
Multi-DLL link errors on an instanceThe global extension handles EXTERNAL/export automatically via %DefaultExternal; make sure the global extension is added and the build's root DLL is set correctly.
Class file changes ignoredYou have two copies on the path. Keep one authoritative copy (e.g. \clarion12\libsrc\win) and re-copy after edits.
